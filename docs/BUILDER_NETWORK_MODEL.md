# Builder Network Model

## Purpose

The La Mesa Builder Network is a vetted pool of builders who can be matched to selected founders and projects.

Builders are not freelancers bidding on random jobs. They are trusted collaborators entering a selective startup formation environment.

## Positioning

Public language:
- Join the La Mesa Builder Network.
- Build with serious founders.
- Turn your skills into startup momentum.
- Earn reputation by helping teams ship.

## Builder Access Model

Builders can create accounts and submit profiles.

Builders cannot view project opportunities before approval.

Once approved, builders become eligible for matching and may later see selected project opportunities based on track, fit, and availability.

## Builder Application

Builder profiles should capture:
- Full name
- Email
- Location
- Languages
- Skills
- Experience level
- Preferred roles
- Portfolio, GitHub, LinkedIn
- Interests
- Availability per week
- Collaboration preference
- Project goals
- Short bio
- Track interests
- Work style
- Reliability signals over time

## Builder Roles

Supported roles:
- Founder / Project Lead
- Full-Stack Developer
- Frontend Developer
- Backend Developer
- AI Engineer
- Blockchain Developer
- UX/UI Designer
- Product Manager
- Marketing/Growth
- Sales/Business Development
- Content Creator
- Operations Coordinator
- Data Analyst
- Researcher
- No-Code Builder

## Review Criteria

ISD should review builders based on:
- Skill evidence
- Portfolio quality
- Availability
- Communication clarity
- Project interests
- Track alignment
- Collaboration preference
- Professionalism
- Reliability signals
- Ability to contribute within 30 days

## Builder Status Model

- Account created
- Profile submitted
- Pending review
- Approved
- Not approved
- Eligible for matching
- Assigned
- Active builder
- Alumni builder

## Matching Criteria

AI and admins should consider:
- Skills
- Preferred roles
- Availability
- Interests
- Role fit
- Project category alignment
- Builder goals
- Track identity
- Work expectations
- Reliability signals

## Reliability Signals

Reliability should become visible over time.

Early admin-facing signals:
- Responsiveness
- Meeting attendance
- Milestone completion
- Founder feedback
- Admin feedback
- Follow-through
- Communication quality

Future participant-facing badges:
- Table 01 Builder
- Demo Day Builder
- Founder Recommended
- Track Specialist
- Shipped Prototype
- Repeat Builder
- High Reliability

## Reputation Philosophy

Builders should not be reduced to ratings.

The goal is to help serious builders earn trust through contribution, not create a shallow ranking system.

Reputation should reflect:
- Work quality
- Reliability
- Team contribution
- Learning speed
- Founder trust
- Demo outcomes

## Builder Visibility

Before approval:
- Builders see their own profile and review status.
- Builders do not see projects.

After approval:
- Builders can see assigned projects.
- In later versions, builders may see selected opportunities or express interest.

Future:
- Builders may receive recommended projects.
- Builders may express interest in projects.
- Founders may shortlist builders.
- Admin remains final approver.

## AI Role

AI should act as:
- Builder profile analyst
- Role fit evaluator
- Project match analyst
- Reliability pattern assistant over time

AI should not:
- Approve builders automatically
- Assign builders automatically
- Make compensation or equity recommendations
- Replace admin judgment
