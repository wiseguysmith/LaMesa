# Builder Network Model

## Purpose

The La Mesa Builder Network is a vetted pool of builders who can support accepted Founder 12 ventures.

Builders are not freelancers bidding on random jobs. They are trusted collaborators entering an ISD-backed founder cohort environment.

## Positioning

Public language:
- Join the La Mesa Builder Network.
- Build with accepted Founder 12 ventures.
- Turn your skills into startup momentum.
- Support founders moving toward prototype readiness.
- Earn trust by helping serious teams ship.

## Builder Access Model

Builders can create accounts and submit profiles.

Builders cannot view venture opportunities before approval.

Approved builders can be considered for admin-guided matching.

For MVP:
- Builders do not browse all ventures.
- Builders do not message founders natively.
- Builders only see assigned Founder 12 ventures.

## Builder Application

Builder profiles should capture:
- Full name
- Email
- Location
- Languages
- Skills
- Experience level
- Preferred roles
- Portfolio, GitHub, LinkedIn
- Interests
- Availability per week
- Collaboration preference
- Project goals
- Short bio
- Work style
- Reliability signals over time

## Builder Roles

Supported roles:
- Founder / Project Lead
- Full-Stack Developer
- Frontend Developer
- Backend Developer
- AI Engineer
- Blockchain Developer
- UX/UI Designer
- Product Manager
- Marketing/Growth
- Sales/Business Development
- Content Creator
- Operations Coordinator
- Data Analyst
- Researcher
- No-Code Builder

## Review Criteria

ISD should review builders based on:
- Skill evidence.
- Portfolio quality.
- Availability.
- Communication clarity.
- Project interests.
- Collaboration preference.
- Professionalism.
- Reliability signals.
- Ability to contribute within a focused build cycle.
- Fit with Founder 12 venture needs.

## Builder Status Model

- Account created.
- Profile submitted.
- Pending review.
- Approved.
- Not approved.
- Eligible for matching.
- Assigned.
- Active builder.
- Alumni builder.

## Matching Criteria

AI and admins should consider:
- Skills.
- Preferred roles.
- Availability.
- Interests.
- Role fit.
- Venture category alignment.
- Builder goals.
- Work expectations.
- Reliability signals.
- Founder 12 venture needs.

## Reliability Signals

Reliability should become visible over time.

Early admin-facing signals:
- Responsiveness.
- Meeting attendance.
- Milestone completion.
- Founder feedback.
- Admin feedback.
- Follow-through.
- Communication quality.

Future participant-facing badges:
- Founder 12 Builder.
- Demo Day Builder.
- Founder Recommended.
- Shipped Prototype.
- Repeat Builder.
- High Reliability.

## Reputation Philosophy

Builders should not be reduced to ratings.

The goal is to help serious builders earn trust through contribution, not create a shallow ranking system.

Reputation should reflect:
- Work quality.
- Reliability.
- Team contribution.
- Learning speed.
- Founder trust.
- Demo outcomes.

## Builder Visibility

Before approval:
- Builders see their own profile and review status.
- Builders do not see ventures.

After approval:
- Builders can see assigned ventures.

Future:
- Builders may receive recommended ventures.
- Builders may express interest in selected opportunities.
- Founders may shortlist builders.
- Admin remains final approver.

## AI Role

AI should act as:
- Builder profile analyst.
- Role fit evaluator.
- Venture match analyst.
- Reliability pattern assistant over time.

AI should not:
- Approve builders automatically.
- Assign builders automatically.
- Make compensation or equity recommendations.
- Replace admin judgment.
